**日期**：2026-06-25

## 📌 TL;DR（执行摘要，3-5 行）

- **核心目标**：打造一款面向游戏开发团队的节点式可视化流程编辑器（代号 FlowForge），以 ComfyUI 级极简画布体验解决游戏开发中剧情/演出/任务/动作/效果等流程配置的"多表联动+配置不可视+协作冲突"三大痛点
- **关键决策**：MVP 优先覆盖剧情编辑+任务编辑（策划60%日常工作），以"配置与实现完全分离"为核心架构，JSON文本格式为数据契约，独立编辑器+引擎Runtime插件为产品形态
- **市场定位**：主攻 AA 级中型团队（20-80人），开源自研核心 + 商业增值，跨引擎（Unity→Unreal→Godot）逐步覆盖
- **下一步**：Phase 1 MVP 9-10 个月，需确认 Tauri/Electron 技术选型、开源协议、首批合作引擎和团队

## 1. 产品目标（3 个清晰、正交的目标）

### 目标一：将"策划-程序"需求传递效率提升 60% 以上

- **当前基线**：策划-程序需求传递平均需 3-5 轮往返（用户研究数据），纯沟通成本占开发周期 25-40%
- **目标值**：通过可视化流程编辑器将需求传递轮次压缩至 1-2 轮，配置产出即为可执行规范
- **度量方式**：Alpha 测试中追踪 10+ 对策划-程序组合的协作轮次，对比历史基线
- **时间节点**：MVP Phase 1 发布后 3 个月内

### 目标二：覆盖中型游戏团队 60%+ 的日常流程配置场景

- **基准数据**：剧情编辑（综合分 65）+ 任务编辑（综合分 63）覆盖约 45% 策划日常工时；演出编辑（44）和动作编辑（36）进一步覆盖 25%
- **目标值**：Phase 1（剧情+任务）覆盖基础场景；Phase 2（+演出+协作+模板）覆盖 60%+；Phase 3（+动作/效果+AI）覆盖 85%+
- **度量方式**：Beta 用户工作日志抽样

### 目标三：建立跨引擎可移植的"配置-实现"分离标准

- **目标值**：一套 JSON Schema 标准 + 至少 2 个引擎 Runtime 插件（Unity + Unreal）+ 1 个开源运行时参考实现
- **度量方式**：JSON Schema 独立仓库，社区贡献者可直接手写/解析工作流文件
- **时间节点**：Phase 1 同步发布 JSON Schema v1.0 + Unity Runtime；Phase 2 扩展至 2+ 引擎

## 2. 用户故事（3-5 个场景）

### 故事 1：剧情策划 — 从 Excel 地狱到可视化叙事

> **作为**一位不会编程的纯剧情策划，**我希望**在无限画布上拖拽"对话节点""分支节点""条件节点"并连线，**以便**能像画流程图一样编排剧情分支，无需在 5 张 Excel 表之间跳转关联。

**验收关键点**：右键搜索拖入节点 / Pin 自动类型匹配 / 属性面板即时编辑 / 一键导出 Yarn Spinner 格式 / JSON 文件 Git 版本对比

### 故事 2：任务策划 — 用节点组封装可复用的任务模板

> **作为**一位懂基础脚本的技术策划，**我希望**将"接任务→杀怪→收集→交任务"封装为可复用节点组，**以便**修改几个参数就能批量生成 50 个同类型任务。

**验收关键点**：框选→右键创建节点组→定义参数 / 节点库中展示 / 双击进入组内编辑 / 参数化暴露属性 / 版本历史可回滚

### 故事 3：程序/技术美术 — 定义领域专用节点类型

> **作为**一位游戏客户端程序员，**我希望**用 DSL 描述文件定义节点 Pin 和属性面板，**以便**策划使用该节点时自动获得类型安全的连线和参数校验。

**验收关键点**：JSON/YAML 节点定义模板 / 自动出现在搜索列表 / Runtime 解析同一份 JSON / 通过插件热加载

### 故事 4：协作场景 — 两人同时编辑不冲突

> **作为**一个 30 人团队的两个剧情策划，**我希望**能同时编辑同一剧情的不同子图（如"第一章"和"第二章"），**以便**合入主线时不会互相覆盖。

**验收关键点**：节点组天然作为协作边界 / 主工作流通过 `$ref` 引用子图文件 / 稳定 UUID 确保 Git 可自动合并 / 内置 diff 视图

### 故事 5：独立开发者 — 快速从模板开始

> **作为**一位使用 Godot 的独立开发者，**我希望**从模板库一键加载"标准对话系统"模板，**以便**10 分钟内完成基础对话系统的流程搭建。

**验收关键点**：模板浏览器社区贡献 / 自动适配引擎导出格式 / 依赖检查提示 / 免费版有规模限制

**vs UE Blueprints**：我方胜在引擎独立、配置分离、Git友好；对方胜在调试能力、生态系统、成熟度。定位："不锁定引擎的Blueprints"。

1. **"配置与实现完全分离"作为第一性原理**
2. **跨引擎可移植**（Unity/Unreal/Godot 插件化 Runtime）
3. **ComfyUI 式极简画布体验 × 游戏领域打磨**
4. **领域专用节点（DSL Nodes）**——不做通用可视化编程

## 7. 关键流程图 / 架构设计

### 7.1 核心架构：配置与实现完全分离

```
┌─────────────────────────────────────────────────────────────┐
│                    配置端（编辑器）                            │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐ │
│  │ 可视化画布 │   │ 节点库    │   │ 属性面板  │   │ 模板浏览器 │ │
│  │(ComfyUI式)│   │(DSL定义) │   │(Markdown)│   │          │ │
│  └─────┬─────┘   └─────┬────┘   └─────┬────┘   └─────┬────┘ │
│        └───────────────┴──────────────┴──────────────┘       │
│                              │                               │
│                    ┌─────────▼─────────┐                     │
│                    │  JSON 工作流文件   │ ◄── Git 仓库         │
│                    │ (FlowForge Schema │     (版本控制/Diff)  │
│                    │  v1.0)            │                     │
│                    └─────────┬─────────┘                     │
└──────────────────────────────┼──────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────┐
│                    实现端（游戏引擎）                           │
│  ┌──────────────┐  ┌───────────────┐  ┌──────────────┐      │
│  │ Unity Plugin │  │ Unreal Plugin │  │ Godot Plugin │      │
│  │ (C# Runtime) │  │ (C++ Runtime) │  │ (GDScript)   │      │
│  └──────┬───────┘  └───────┬───────┘  └──────┬───────┘      │
│  ┌──────▼──────────────────▼──────────────────▼───────┐      │
│  │        FlowForge Runtime Core                      │      │
│  │  解析器(Parser) │ 执行器(Runner) │ 变量存储 │ 事件总线 │      │
│  └────────────────────────────────────────────────────┘      │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 数据流

```
策划拖拽节点 → 连线+填写属性 → 保存JSON工作流 → Git提交
→ 程序拉取JSON → Runtime解析 → 驱动游戏逻辑
→ 策划修改JSON → 不需重新编译
```

### 7.3 节点类型系统

**Pin 类型**：flow（执行流）、string、number、boolean、any、array、reference

**连线规则**：同类型可互连 / flow只连flow / any连任意但运行时类型检查 / 输入Pin默认1连线 / 输出Pin可多连线

### 7.4 子图引用机制（协作基础）

主工作流 `main_quest_flow.json` 通过 `$ref` 引用子图文件 `./chapters/chapter1.json`。不同策划独立编辑不同子图文件，Git合并无冲突。

## 8. Non-goals（明确不做什么）

| 不做的方向 | 理由 | 替代方案 |
|-----------|------|---------|
| 不做通用可视化编程（替代Blueprints） | 与引擎内置工具直接竞争，复杂度失控 | 提供"脚本节点"嵌入Lua/Python |
| 不做全生命周期内容管理（替代Articy） | 聚焦"流程配置"单一职责，避免第二系统效应 | 提供数据源连接器 |
| 不做实时协作编辑（Phase 1-2） | 技术复杂度极高（OT/CRDT） | Git协作模式 + 子图文件引用 |
| 不做引擎内置工具 | 引擎独立是核心差异化 | 独立桌面应用 + Runtime插件 |
| 不做纯SaaS/不强制云服务 | 游戏团队对数据安全和离线有强需求 | 本地优先，Git原生，云服务可选 |
| 不做替代专业写作工具 | 文本创作是独立专业领域 | 支持Markdown和Yarn Spinner导出 |
