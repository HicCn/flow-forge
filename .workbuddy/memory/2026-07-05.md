## 架构决策：流间传参 + Tab 系统 + flow_call 节点

### 决策结论
- **流间传参必要性**：需要，但不急。正确时机是 P1（Subgraph/$ref 实现前）。MVP 只做架构预留。
- **传参方式**：键值对。参数绑定是**编辑时行为**（不是运行时），Runtime 不参与。编辑器读取目标流的 `isInput` 参数列表，策划在 flow_call 节点面板填入值。
- **入参标记方案**：在现有 Parameters 面板上加 `isInput` 复选框（非新增独立的 interface 栏目），避免二次赋值。

### 技术要点
- Tab 数据存储在 editorStore，切换时 flush active tab 再 load 目标 tab
- `openTab` 自动从 `flowType` 推导 `nodeDefinitions`，调用方无需手动传入
- flow_call 节点的目标流接口参数通过 `loadFile` 延迟读取（先查现有 tabs，再读磁盘）
- 参数 isInput 的 visual indicator：标签从类型缩写变为 `in`，颜色从 info 变为 warning
