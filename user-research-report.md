## 一、用户画像

### 1.1 核心用户群体总览

| **系统策划（System Designer）** | 系统架构、数值平衡、规则设计 | ★★★☆☆（中等） | Excel（大量配表）、内部配置工具 | 任务链设计、技能参数配置、数值调优 |

### 1.2 用户分层画像

#### 第一层：纯策划型用户（占比 ~45%）
- **代表角色**：剧情策划、初级关卡策划、系统策划
- **核心特征**：
  - 不具备编程能力，依赖可视化工具
  - 日常工作重度依赖 Excel 配表（每周可能编辑 50-200+ 张表）
  - 对「节点 + 连线」范式接受度高（类 ComfyUI 的直觉操作）
  - 核心痛点：配置表之间的关联关系不可视，改一处漏多处
- **工具偏好**：拖拽式、所见即所得、内置校验、模板驱动

#### 第二层：技术策划型用户（占比 ~30%）
- **代表角色**：技术策划、战斗策划、高级关卡策划
- **核心特征**：
  - 具备基础脚本/蓝图能力，理解状态机和事件驱动模型
  - 需要工具既能可视化编排，又能暴露底层数据细节
  - 经常扮演策划与程序的「翻译官」角色
  - 核心痛点：工具要么太简单（无法表达复杂逻辑），要么太复杂（需要写代码）
- **工具偏好**：可扩展节点系统、支持自定义节点、混合编辑（可视化+脚本）

#### 第三层：程序/实现端用户（占比 ~25%）
- **代表角色**：Gameplay 程序、工具链程序、引擎程序
- **核心特征**：
  - 不直接使用编辑器做日常配置，但需要：
    - 导入策划产出的配置数据
    - 开发新的节点类型供策划使用
    - 维护配置到运行时的数据管线
  - 核心痛点：策划产出数据不规范、格式多变、导入导出反复调试
- **工具偏好**：标准化导出格式、类型安全、版本可追踪、CI/CD 集成

## 二、核心场景与使用流程

### 2.1 场景一：剧情编辑

**典型流程**：
```
[创建剧情图] → [添加对话节点] → [编排分支选项] → [设置条件变量] 
→ [预览对话流] → [导出/对接引擎]
```

**关键节点类型**：
- **对话节点**：说话人、文本内容、表情/姿态标记
- **选择节点**：玩家可选分支（2-5个选项常见）
- **条件节点**：基于变量判断（好感度、任务进度、持有道具等）
- **跳转节点**：跨场景/跨对话跳转
- **事件节点**：触发动画、音效、镜头切换、奖励发放

**复杂度特征**：
- 典型 RPG：500-2000 个对话节点
- AAA 项目：5000-20000+ 个节点
- 分支深度通常 3-5 层，但有大量平行分支
- 复用场景多：同角色对话模板、通用对话片段

**现有解决方案对比**：

| 方案 | 优势 | 劣势 |
|------|------|------|
| **Excel/CSV** | 简单，程序友好 | 分支不可视，关联难追踪，diff 困难 |
| **Twine** | 免费，简单上手，Web 可用 | 不集成引擎，无版控，仅原型 |
| **Ink/Yarn Spinner** | 轻量脚本，Unity 集成好 | 无可视化编辑，需要书写脚本语法 |
| **Articy:Draft X** | 全功能，工业级，引擎集成 | 贵（5.99-49€/月），Windows 为主，学习曲线陡 |
| **Arcweave** | 浏览器协作，低门槛 | 无运行时引擎，需自行集成 |

**本产品机会**：提供「类 ComfyUI」无限画布的叙事编排体验，填补免费/低价可视化工具与工业级 Articy 之间的空白。

---

### 2.2 场景二：演出编辑

**典型流程**：
```
[创建演出序列] → [添加镜头机位] → [编排时间轴] → [添加动画/音效事件] 
→ [设置过渡效果] → [预览/回放] → [导出 Timeline 数据]
```

**关键节点类型**：
- **镜头节点**：机位、FOV、景深、运动路径
- **时间轴节点**：时间刻度、关键帧标记
- **动画触发节点**：角色动画、物体动画、Blend 参数
- **音效节点**：BGM 切换、音效触发、音量渐变
- **特效节点**：粒子效果、后处理、屏幕震动
- **过渡节点**：淡入淡出、切镜方式、持续时间
- **分支节点**：条件触发不同演出路线

**行业参考**：
- Unity Timeline + CineGraph（节点式过场编辑器，2025年发布）
- UE Sequencer（时间轴为主，非纯节点式）
- 自研方案为主（大厂基本都有内部过场编辑器）

**核心痛点**：
- 时间轴与节点两种范式的切换成本
- 多部门协作：策划定节奏 → 美术做镜头 → 程序接事件
- 预览迭代慢：改一个参数需要重新进游戏验证

---

### 2.3 场景三：任务编辑

**典型流程**：
```
[创建任务图] → [定义任务节点] → [设置前置条件] → [配置任务目标] 
→ [设置奖励/后续] → [关联对话/战斗等] → [导出任务数据]
```

**关键节点类型**：
- **任务节点**：名称、描述、类型（主线/支线/日常）
- **条件节点**：等级限制、前置任务、持有道具、NPC 状态
- **目标节点**：击杀 N 个怪物、收集道具、到达地点、对话完成
- **奖励节点**：经验、金币、道具、解锁内容
- **分支节点**：任务完成/失败的不同后果链
- **并行节点**：可同时进行的多个目标

**复杂度特征**：
- 中等 RPG：100-500 个任务
- AAA 开放世界：500-3000 个任务
- 任务间依赖关系图复杂（DAG 结构）
- 条件组合多样：AND/OR/NOT 逻辑嵌套

**现有方案**：
- 大多依赖 Excel 配置（任务表、任务条件表、任务奖励表...多表关联）
- 少数使用自研可视化任务编辑器（如 Grasscutter Quest Editor）
- LoomGraph、QuestGraph（UE 插件）提供节点式任务编排

---

### 2.4 场景四：动作编辑

**典型流程**：
```
[创建技能/动作图] → [定义状态节点] → [编排状态转换] → [配置触发条件] 
→ [设置动作参数] → [预览状态机] → [导出技能配置]
```

**关键节点类型**：
- **状态节点**：待机、攻击、受击、跳跃、闪避等
- **转换节点**：状态转换条件（输入、时间、碰撞等）
- **动作片段节点**：动画 Clip、持续时间、混合参数
- **判定节点**：伤害判定框、击退参数、硬直时间
- **组合节点**：连招序列、分支连击
- **效果节点**：屏幕震动、定帧、子弹时间、Buff 附加

**行业参考**：
- UE 状态机 + Gameplay Ability System
- Unity Animator Controller + PlayMaker / NodeCanvas
- 格斗游戏常用自研连招编辑器（节点式组合技编排）

**核心痛点**：
- 状态机可视化程度高但逻辑配置（如伤害计算）仍需分离
- 策划调整一个参数（如硬直时间）需要在多处同步
- 动作预览需要进游戏，反馈周期长

---

### 2.5 场景五：效果编辑

**典型流程**：
```
[创建效果图] → [添加特效节点] → [编排效果序列] → [配置参数曲线] 
→ [设置触发条件] → [实时预览] → [导出效果配置]
```

**关键节点类型**：
- **发射器节点**：粒子生成规则、发射速率、生命周期
- **行为节点**：速度、旋转、缩放、颜色变化
- **碰撞节点**：碰撞检测、反弹、销毁
- **渲染节点**：材质、贴图、混合模式
- **后处理节点**：Bloom、DOF、Color Grading、Motion Blur
- **音效联动节点**：特效与音效的同步触发
- **曲线节点**：Float/Curl/Color 曲线参数

**行业参考**：
- Unity VFX Graph（节点式粒子编辑器，业界标杆）
- UE Niagara（模块化特效系统，非纯节点式但理念相近）
- Blender Geometry Nodes（3D 建模/特效节点式系统）

**思考**：效果编辑已有较成熟的节点式方案（VFX Graph），本产品的差异化在于——让策划/TA 能在一个统一的节点式环境中编排跨系统的效果触发逻辑（如「Boss 进入二阶段 → 播放特效序列 + 镜头震动 + BGM 切换」），而非替代专业 VFX 工具。

## 三、用户痛点分析

### 3.1 现有方案痛点矩阵

| 痛点类别 | 具体表现 | 影响面 | 严重度 |
|---------|---------|--------|--------|
| **配置不可视** | Excel 表中几千行数据，关联关系完全靠人工记忆和注释 | 全角色 | 🔴 极高 |
| **多表联动难** | 改一张表需要同步改 3-5 张关联表，漏改 = Bug | 策划 > 程序 | 🔴 极高 |
| **协作冲突频发** | 多人同时编辑同一张配置表，合并时出错，数据覆盖 | 全角色 | 🔴 极高 |
| **校验靠人工** | 配置错误（引用不存在的 ID、条件逻辑死循环）只能在游戏运行时发现 | 策划 > QA | 🟠 高 |
| **版控困难** | Excel 二进制 diff 困难，节点图 merge 几乎不可能 | 全角色 | 🟠 高 |
| **需求传递失真** | 策划「这份配置的意思是...」→ 程序理解偏差 → 返工 | 策划 ↔ 程序 | 🟠 高 |
| **缺乏复用机制** | 相似配置（如通用对话模板）反复创建，一次修改需到处同步 | 策划 | 🟡 中高 |
| **工具碎片化** | 剧情用 Twine，任务用 Excel，动作用状态机，数据不互通 | 全角色 | 🟡 中高 |
| **学习成本高** | Articy 功能强大但学习曲线 2-4 周，非专业团队难以承受 | 小型团队 | 🟡 中 |

### 3.2 策划-程序协作摩擦深度分析

**核心矛盾**：策划的「需求语言」与程序的「实现语言」之间存在翻译层，而当前的中间格式（Excel/文档）不能有效承载这个翻译层。

**典型摩擦链路**：
```
策划设计意图 → 写成需求文档 → 填入 Excel 配置 
→ 程序读表理解 → 编写实现代码 → 运行时测试 
→ 发现理解偏差 → 沟通修正 → 改表/改代码 → 重复
```

**统计参考**（行业经验 + 文献综合）：
- 配置-实现往返次数：平均 3-5 轮
- 单次配置变更沟通成本：30min - 2h
- 因配置问题导致的 Bug 占比：中小团队 20-35%
- 「2 天工作量缩短至 2 小时」是真实存在的效率差距（引自知乎行业案例）

**本产品解决的协作命题**：
- 配置端提供 **结构化、类型化、可校验** 的节点定义
- 需求在配置阶段就被「形式化」，减少理解偏差
- 节点 = 需求的最小契约单元：策划定义「需要什么」，程序实现「怎么做到」
- 实现「需求前置」：配置完成时就已经完成了逻辑验证

### 3.3 Excel 地狱：典型案例

> 某中型 RPG 项目的任务配置体系：
> - 27 张 Excel 表（任务主表、条件表、奖励表、对话映射表、NPC 表...）
> - 每张表 50-2000 行
> - 表间通过 ID 外键关联（完全不可视）
> - 一次新增任务类型需要改 8 张表
> - 每张表都多人编辑，每周至少 2-3 次合并冲突
> - 新人上手需要 2-3 周才能独立配任务

**在节点式编辑器中的等效体验**：
- 一张无限画布上的任务图
- 节点颜色区分类型，连线表示依赖
- 右键新建任务节点 → 连接前置条件 → 配置目标 → 设置奖励 → 完成
- 自动检测循环依赖、缺失引用、条件冲突
- 一键导出结构化数据给程序

## 四、用户期望

### 4.1 理想节点式编辑体验

#### 核心理念
以 ComfyUI 为代表的新一代节点编辑器定义了以下体验标准，应作为设计基准：

| 体验维度 | ComfyUI 模式 | 对本产品的启发 |
|---------|-------------|---------------|
| **无限画布** | 自由缩放、平移，无边界画布 | 支持大型任务图/剧情图，不设节点数限制 |
| **拖拽连线** | 直观的端口拖拽建立连接 | 一致的交互模式，降低学习成本 |
| **即时反馈** | 节点状态实时显示（执行中/完成/错误） | 配置校验即时反馈，错误节点高亮 |
| **节点库** | 右侧可搜索节点面板 | 按场景分类的节点库（剧情/任务/动作/效果） |
| **子图/模块化** | 节点组折叠/展开 | 支持节点组封装和复用（如「通用对话模板」） |
| **快照/版本** | 工作流保存与版本切换 | 内置版本历史，支持 diff 和回滚 |

#### 场景特定的体验期望

**剧情编辑场景**：
- 🎯 **对话预览模式**：在编辑器中直接模拟对话流程
- 🎯 **分支覆盖检查**：高亮未覆盖的分支路径
- 🎯 **文本版本管理**：支持文案迭代而不影响逻辑结构
- 🎯 **本地化映射视图**：在一个界面查看所有语言的文本映射

**演出编辑场景**：
- 🎯 **双模式切换**：节点图 ↔ 时间轴视图一键切换
- 🎯 **实时预览**：在编辑器中回放演出序列（需要引擎集成）
- 🎯 **镜头书签**：保存常用镜头预设为可复用节点

**任务编辑场景**：
- 🎯 **依赖图分析**：可视化任务间的依赖关系、关键路径
- 🎯 **条件可视化**：AND/OR 逻辑树以嵌套节点或颜色标记展示
- 🎯 **任务链模拟**：模拟玩家完成任务的路径和状态变化

**动作编辑场景**：
- 🎯 **状态机+节点图融合**：状态转换用状态机视图，动作参数用节点图
- 🎯 **动画预览集成**：选中动作节点即可预览对应动画
- 🎯 **数值曲线编辑器**：内嵌曲线编辑器调整动作参数

**效果编辑场景**：
- 🎯 **实时 VFX 预览**：编辑器中实时渲染效果（需引擎集成）
- 🎯 **层级化管理**：效果序列的分层管理（背景层/角色层/UI 层）
- 🎯 **参数驱动**：节点参数支持曲线和关键帧

### 4.2 可视化 vs 文本配置的偏好

**研究发现**：
- 对于**流程/结构类**配置（如剧情分支、任务链、演出序列），可视化节点图 > 文本
- 对于**数据/数值类**配置（如属性值、掉落率、伤害公式），表格/文本 > 节点图
- 技术策划倾向「混合模式」：可视化编排 + 文本脚本嵌入

**推荐设计策略**：

| 配置类型 | 推荐主模式 | 辅助模式 |
|---------|-----------|---------|
| 流程/逻辑编排 | 节点图（可视化） | 节点属性面板（表单） |
| 数值参数调整 | 属性面板/批量表格 | 节点图中数值预览 |
| 条件表达式 | 可视化条件构建器 | 文本表达式（高级用户） |
| 批量数据导入 | CSV/JSON 导入 | 节点图自动生成 |
| 脚本扩展 | 文本编辑器 | 节点外挂脚本 |

### 4.3 节点复用与模块化

**用户期望优先级（从调研中提炼）**：

1. **节点组（Node Group）** 🔴 最高
   - 将一组节点打包为可复用模块
   - 例如：「标准对话模板」（说话人 + 文本 + 选项分支）
   - 修改模板→所有使用处同步更新

2. **预设系统（Preset）** 🟠 高
   - 节点参数的一键填充模板
   - 例如：「常见奖励配置」（经验100+金币50）
   - 用户可创建和分享预设

3. **子图引用（Subgraph Reference）** 🟠 高
   - 跨文件引用子图
   - 例如：「通用 NPC 对话」被多个剧情图引用
   - 修改一处，全局更新

4. **节点市场/库（Node Marketplace）** 🟡 中
   - 团队/社区共享的自定义节点
   - 程序开发新节点类型→策划直接使用

### 4.4 版本管理需求

基于叙事脚本版控研究（PulseGeek, 2026）的关键发现：

| 需求 | 说明 | 优先级 |
|------|------|--------|
| **稳定节点 ID** | 每个节点创建后 ID 不变，保证 diff 可读 | 🔴 必须 |
| **文本格式存储** | JSON/YAML 格式，支持 Git diff 和 merge | 🔴 必须 |
| **可视化 Diff** | 在编辑器中展示两个版本的节点图差异 | 🟠 高 |
| **分支管理** | 允许策划在独立分支上实验性修改，满意后合并 | 🟠 高 |
| **冲突解决** | 多人编辑同一节点时的冲突检测和合并向导 | 🟠 高 |
| **回滚机制** | 一键回退到任意历史版本 | 🟡 中高 |
| **变更注释** | 每次保存要求填写变更说明 | 🟡 中 |

## 五、场景化需求优先级排序

### 5.1 排序方法论

采用「需求紧迫度 × 通用性 × 市场空白度」三维评分模型：

- **需求紧迫度**：该场景在游戏开发中被解决的急迫程度（1-5）
- **通用性**：该场景的节点编辑模式可复用到其他场景的程度（1-5）
- **市场空白度**：现有工具在该场景的覆盖不足程度（1-5）

### 5.2 排序结果

| 排名 | 场景 | 紧迫度 | 通用性 | 空白度 | 综合分 | 核心理由 |
|------|------|--------|--------|--------|--------|---------|
| 🥇 1 | **剧情编辑** | 5 | 4 | 4 | **65** | 使用频率最高、Excel 弊端最突出、现有可视化工具贵/重/不通用 |
| 🥈 2 | **任务编辑** | 5 | 5 | 4 | **63** | 与剧情紧密关联、节点图（DAG）是自然表达、复用价值最高 |
| 🥉 3 | **演出编辑** | 4 | 3 | 4 | **44** | 节点+时间轴混合需求独特、大厂自研为主、中等团队无方案 |
| 4 | **动作编辑** | 4 | 3 | 3 | **36** | 已有较成熟方案（状态机编辑器）、差异化空间在「连招编排」 |
| 5 | **效果编辑** | 3 | 2 | 2 | **18** | VFX Graph/Niagara 已较好覆盖、本产品差异化在跨系统联动 |

### 5.3 推荐 MVP 范围

**Phase 1（MVP）**：剧情编辑 + 任务编辑
- 理由：两个场景共享核心节点图基础设施，协同效应强
- 覆盖 ~60% 的策划日常配置工作
- 难度适中，6-9 个月可交付可用版本

**Phase 2**：演出编辑
- 引入时间轴视图与节点图的双模式
- 需要一定的引擎集成（动画/镜头预览）

**Phase 3**：动作编辑 + 效果编辑
- 受益于前两阶段的基础设施积累
- 面向更专业的细分用户

### 6.3 本产品差异化定位

> **「面向游戏开发团队的统一节点式流程编辑器」—— 一个编辑器，覆盖剧情、任务、演出、动作、效果五大配置场景。以 ComfyUI 级的易用性和无限画布体验，实现「需求前置」——让策划在配置阶段就完成逻辑验证，而非等程序实现后才发现问题。**

核心差异点：
1. **统一性**：一个编辑器覆盖多场景，而非每个场景一个工具
2. **需求前置**：配置即设计，设计即校验，减少策划-程序往返
3. **可扩展**：程序可开发自定义节点类型，策划可直接使用
4. **版本友好**：文本格式存储 + 稳定 ID，Git 友好
5. **ComfyUI 级体验**：无限画布，拖拽连线，即时反馈

## 七、关键建议

### 7.1 产品设计建议

1. **先做节点图引擎再做场景适配**：核心画布、连线、节点系统的架构决定了各场景的上限。建议投入 60% 精力在核心引擎，40% 在场景适配。

2. **存储格式从 Day 1 就支持版本控制**：使用 JSON + 稳定 UUID 作为存储格式，确保 Git diff 可读。这是行业级需求，也是后续协作功能的基础。

3. **节点即契约**：每个节点类型的定义应包含：输入端口类型、输出端口类型、参数 Schema、校验规则。策划在编辑器中看到的每个节点，背后都有程序定义的「功能契约」。

4. **提供「逃生舱」**：可视化编排 85% 的场景，但允许在节点中嵌入文本脚本（Lua/Python 表达式），避免「可视化天花板」。

5. **预览能力是核心竞争力**：对话预览、任务路径模拟、演出回放——这些预览能力是可视化编辑器的「杀手级功能」。

### 7.2 技术架构建议

- **前后端分离**：编辑器 UI 层 + 数据模型层 + 引擎集成层
- **节点类型注册机制**：程序通过插件/注册方式添加新节点类型
- **实时校验引擎**：配置修改时即时运行校验规则（类型检查、引用完整性、逻辑循环检测）
- **导出管线**：支持 JSON/YAML/Protobuf 多种导出格式，可配置导出模板

### 7.3 后续研究建议

1. **用户访谈验证**：选取 3-5 个目标团队（覆盖大型/中小/独立）进行深度访谈，验证本报告的假设
2. **竞品深度试用**：对 Articy、Arcweave、LoomGraph 进行 40+ 小时深度试用评测
3. **工作流观察研究**：在真实项目中观察策划-程序的完整协作链路
4. **MVP 原型测试**：基于本报告的 MVP 建议，快速构建低保真原型进行可用性测试
5. **价格敏感度调研**：了解不同规模团队对工具付费的意愿和预算范围
